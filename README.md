# mini-koa

这个仓库会通过 Git 提交历史，一步步把 mini-koa 搭出来。

目标不是尽快把功能堆满，而是按最适合理解 Koa 的顺序，让同一个核心文件慢慢长出来。

换句话说，这个仓库更在意“为什么下一步应该是它”，而不是“还能不能再多加一个功能”。

## 怎么看这条历史

每一条提交只做一件事，所以这条历史本身就是学习路线。

你可以先看完整代码，再按顺序回看提交，观察同一个 `src/application.js` 是怎么一步步长出来的。

常用方式：

```bash
git log --oneline --reverse
```

如果你想看某一步的具体变化，可以继续用：

```bash
git show <commit-id>
```

如果你想按顺序体验每一步，可以从最早的提交开始，一步步切过去看。

## 为什么要按这个顺序

如果目标是学 Koa 本身，而不是尽快做一个能配很多插件的小 Web 框架，那顺序非常重要。

最容易走偏的方式，是太早去做这些东西：

- 先做路由
- 先做模板
- 先做请求体解析
- 先做一堆工具方法

这样虽然看起来“功能更多”，但会把 Koa 最核心的那条主线打散。

Koa 真正最值得先弄明白的，是这条链：

1. 请求是怎么进来的
2. 应用对象为什么要先出现
3. 中间件为什么必须先能注册
4. 多个中间件为什么会形成洋葱模型
5. 为什么还要再包出一个统一入口
6. 为什么最后还要有统一收尾、统一报错和默认兜底

只有这条主线通了，后面的路由、请求体、插件能力，才知道该挂在哪、为什么能挂上去。

## 推荐完善顺序

这是比较适合学习 Koa 核心的顺序：

1. Node 原生服务
2. Application 外壳
3. `use` 注册单个中间件
4. 多中间件串联和洋葱模型
5. `ctx` 统一入口
6. 异步、默认返回、统一报错
7. 更像 Koa 的代理能力
8. 更完整的返回处理
9. 再考虑路由和请求体解析

这里面最重要的一点是：

路由不是 Koa 核心的起点，而是核心稳定之后，才适合往外接的一层。

## 第一步

先只用 Node 原生 `http` 做一个最小服务。

这一版只解决最基础的问题：

1. 浏览器来请求时，Node 会收到什么
2. 服务最后到底是怎么把内容返回出去的

这一步故意什么都不抽象，只保留最原始的样子。因为后面所有封装，最后都还是要落回这里。

> **为什么下一步不是直接做路由？**

反例：

如果这时就急着加 `/`、`/about`、`/user` 这些分支，你只能把越来越多的判断继续堆在入口文件里。功能是变多了，但结构反而更乱。

因为这时候连“应用对象”都还没有，任何新功能都只能继续堆在入口文件里。先加功能，只会把代码堆得更乱。

## 第二步

把“启动服务”收进 `src/application.js`。

这一版开始做第一层整理。

重点不是功能变多，而是开始有“应用对象”这个概念。以后不管是挂处理中间层、做上下文，还是统一收尾，都有了一个明确归属，不再散落在入口文件里。

这一步主要解决的问题是：

“现在虽然能跑，但所有东西都堆在一个地方，代码没有主人。”

> **为什么下一步是“接收一段处理逻辑”？**

反例：

如果第二步只是把 `listen` 收起来，但“请求来了以后怎么处理”还是写死在应用内部，那它依然只是个壳子，不像一个能被外部驱动的框架。

因为现在只是把服务壳子收起来了，但“请求来了以后到底谁来处理”还写死在应用内部。要想像 Koa，一定要先把处理权交出来。

## 第三步

让应用能接收一段处理逻辑，并由这段逻辑决定不同地址返回什么。

这一版开始靠近 Koa 的味道。

这里最重要的变化不是“地址分流”本身，而是应用终于能把“处理请求这件事”交给外部传进来的函数。只要这件事成立，后面才有可能从一个处理函数，长成很多层处理函数。

这一步主要解决的问题是：

“应用已经有壳了，但还不会接收你自己定义的处理过程。”

> **为什么下一步不是继续加强地址分流？**

反例：

如果这时继续补更多地址判断，看起来像是功能在增长，但本质上还是“一个大函数里塞更多 if”。这会把 Koa 最核心的层级结构完全绕开。

因为 Koa 真正的核心不是“把地址分开”，而是“把处理过程分层”。地址分流只是单层处理里的一个分支判断，不是框架最关键的结构。

## 第四步

继续丰富同一个 `src/application.js`，让它能按顺序串起多层处理，跑出洋葱模型。

这一版是整个项目的第一个关键转折点。

从这里开始，请求不再是“交给一个函数，做完就结束”，而是会按顺序穿过很多层。外层先开始，里层后开始；里层先结束，外层后结束。也就是常说的“先进去，再出来”的洋葱模型。

你如果访问这一版，会看到类似这样的顺序：

```text
1 start -> 2 start -> 3 start -> 4 start -> 5 -> 4 end -> 3 end -> 2 end -> 1 end
```

这一步最值得盯住的不是输出内容，而是这个顺序为什么会出现。

这一步主要解决的问题是：

“一个处理中间层不够时，应用怎么把很多层处理串起来，而且还能先进去、再出来？”

> **为什么下一步不是直接补更多中间件工具？**

反例：

如果洋葱模型刚形成，就立刻往上堆很多工具方法，那每一层依然得直接碰底层对象。看起来“能力更多”，写起来却还是很碎。

因为虽然多层处理已经成立了，但每一层还都在直接碰底层对象。结构有了，使用方式却还很粗糙。

## 第五步

继续在同一个 `src/application.js` 上增加统一入口，让中间件通过 `ctx` 读取请求信息并设置返回内容。

这一版解决的是“为什么不要每一层都直接碰原始对象”。

如果一直直接用底层对象，代码会越来越碎。统一入口出来以后，每一层中间处理都围着同一个东西工作：

- 从这里读地址
- 从这里读请求方式
- 从这里设置返回内容

这样后面继续长能力时，就不需要把细节散在一堆地方。

这一步主要解决的问题是：

“虽然已经有洋葱模型了，但每一层都还在直接操作底层对象，写起来很碎，也不利于继续扩展。”

> **为什么下一步要补异步、报错和兜底？**

反例：

如果有了 `ctx` 就停住，一旦某层异步稍微慢一点、忘了返回内容、或者直接抛错，整个请求的收尾立刻就会变得不稳定。

因为到了这一步，结构和统一入口都已经有了。接下来最真实的问题是：真正跑起来后，空返回怎么办，异步怎么办，抛错怎么办。

## 第六步

继续在同一个 `src/application.js` 上增加更稳的收尾能力。

这一版主要补三件事：

1. 异步流程要能自然走通
2. 没有返回内容时，要有默认兜底
3. 中间层里抛出来的错误，要能被统一接住并转成清楚的返回

你可以重点试这几个地址：

- `/`：正常返回
- `/slow`：异步等待后再返回
- `/error`：故意抛错，看统一处理
- `/missing`：什么都不返回，看默认兜底

运行：

```bash
npm start
```

这一步主要解决的问题是：

“框架现在能跑，但还不够稳。只要一碰到异步、空返回或异常，就会暴露出收尾不统一的问题。”

> **为什么第七步更适合补代理能力？**

反例：

如果第六步之后直接去做路由或请求体解析，那你会发现统一入口虽然存在，但用起来还不够顺手。很多信息还是得绕回更底层去拿。

因为现在已经有了 `ctx`、也有了统一收尾。接下来最值得做的，不是立刻去接更多外围功能，而是把 `ctx.url`、`ctx.method` 这类体验继续打磨得更像真正的 Koa。

## `ctx`、`request`、`response`、`req`、`res` 到底是什么关系

这部分第一次看最容易懵。

尤其是看到 `request.js` 里直接写 `this.req.url` 时，很容易觉得：

“这个 `req` 是从哪里冒出来的？”

这里要分清楚两组东西：

1. Node 原生对象
2. 我们自己包出来的对象

Node 原生给的是：

- `req`
- `res`

我们自己包出来的是：

- `ctx`
- `request`
- `response`

它们的关系可以先记成这样：

```text
ctx
├── request
│   └── req   (Node 原生 request)
└── response
    └── res   (Node 原生 response)
```

也可以理解成：

- `req` 和 `res` 是最底层原料
- `request` 和 `response` 是我们对原料做的薄包装
- `ctx` 是再往上一层的统一入口

所以 `request.js` 里并不是平白无故就有了 `req`。

真正把它挂上去的地方在 `application.createContext(...)` 里。

每次请求进来时，都会临时做这几件事：

1. 先创建一个新的 `ctx`
2. 再创建一个新的 `request`
3. 再创建一个新的 `response`
4. 把 Node 原生的 `req` 挂到 `request.req`
5. 把 Node 原生的 `res` 挂到 `response.res`
6. 再把这几个对象互相连起来

所以这时：

- `request.url` 实际上读的是 `request.req.url`
- `ctx.url` 实际上又继续读的是 `ctx.request.url`

最后中间件只需要面对 `ctx`，但它背后其实还是一路走到了 Node 原生对象。

换句话说，这不是“变量没定义”，而是“定义发生在别的地方”。

如果只单看 `request.js`，确实会觉得跳。要把它和 `application.js` 里的上下文创建过程一起看，整条链才完整。

## 第七步建议做什么

如果按“最像 Koa 核心”的顺序继续往下，下一步我建议做：

`ctx`、`request`、`response` 之间更完整的代理关系。

这一步会主要解决：

1. 为什么 `ctx` 上能直接拿到更多信息
2. 为什么有些能力属于 `request`
3. 为什么有些能力属于 `response`
4. 为什么最后又想让使用的人大多数时候只面对 `ctx`

做完这一步，mini-koa 的核心骨架就会更完整。再往后去接路由、请求体解析，逻辑就会顺很多。

## 第七步

第七步就做这件事：把 `ctx`、`request`、`response` 之间的代理关系再补完整一点。

这一版主要增加这些体验：

- 可以直接用 `ctx.path`
- 可以直接用 `ctx.querystring`
- 可以直接用 `ctx.query`
- 可以直接用 `ctx.get(...)` 读取请求头
- 可以直接用 `ctx.set(...)` 设置响应头
- 可以直接用 `ctx.type` 指定返回类型

这一步主要解决的问题是：

“虽然已经有了统一入口，但这个入口还不够顺手。很多信息还是要绕到更底层去拿，很多返回细节也还不够像真正的 Koa。”

你可以重点试这几个地址：

- `/`：看 `ctx.path`、`ctx.get(...)`
- `/query?name=mini-koa&step=7`：看查询参数如何被代理出来
- `/headers`：看自定义响应头和返回类型
- `/slow?from=readme`：看异步下这些代理依然正常
- `/error`：看错误处理依然保持住

> **为什么这一步在路由前面？**

反例：

如果统一入口还不够顺手就急着接路由，后面写路由时还是会频繁地往下钻，到处手动拿路径、头信息和返回设置。

因为这一步是在打磨 Koa 自己的手感。只有统一入口足够顺，后面再去挂路由、请求体解析这些外围能力，使用体验才会自然。

## 第八步

第八步继续往“统一收尾”这条线上补，不是再加一个新入口，而是把不同类型的返回处理得更完整。

这一版主要增加这些能力：

- 字符串返回会自动补上长度
- 对象返回会自动转成 JSON
- Buffer 返回会按二进制方式处理
- 流式返回会直接流出去
- `204` 这类本来就不该有内容的返回，会自动清掉内容相关头信息

这一步主要解决的问题是：

“虽然现在已经能统一收尾了，但不同类型的返回还没有被认真区分。文本、对象、二进制、流式数据，本来就不该用同一种方式草草结束。”

你可以重点试这几个地址：

- `/`：普通文本返回
- `/query?name=mini-koa&step=8`：JSON 返回
- `/buffer`：二进制返回
- `/stream`：流式返回
- `/empty`：204 空返回
- `/error`：错误返回

> **为什么这一步必须在路由前面？**

反例：

如果现在直接去做路由，你只是把“请求分到哪里”解决了，但“分到那里以后到底怎么稳定返回”还是粗糙的。这样只会更快暴露出返回层的不完整。

所以第八步要先把“返回内容怎么收尾”打磨完整。只有这一层足够稳，后面再加路由，请求到了不同地方以后，才能自然地返回文本、JSON、Buffer 或空内容。
